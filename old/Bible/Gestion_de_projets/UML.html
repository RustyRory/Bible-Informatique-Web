UML

UML (Unified Modeling Language) est un langage de modélisation graphique utilisé en développement logiciel pour visualiser, spécifier, construire et documenter les systèmes. Voici un guide détaillé sur les principaux diagrammes UML avec des exemples concrets.

1. Diagramme de cas d'utilisation

Le diagramme de cas d'utilisation représente les interactions entre les utilisateurs (acteurs) et le système.

Exemple concret : Système de bibliothèque 

graph TD
    A[Lecteur] -->|Emprunte un livre| B((Système de bibliothèque))
    A -->|Retourne un livre| B
    C[Bibliothécaire] -->|Gère les livres| B
    C -->|Gère les membres| B

2. Diagramme de classes

Le diagramme de classes montre la structure statique du système, décrivant les classes, leurs attributs, méthodes et relations.

Exemple concret : Système de gestion d'une école 

classDiagram
    class Etudiant {
        -int id
        -String nom
        -String prenom
        +void s'inscrire()
        +void passerExamen()
    }
    class Cours {
        -int code
        -String titre
        +void ajouterEtudiant()
        +void donnerNote()
    }
    class Professeur {
        -int id
        -String nom
        +void enseignerCours()
        +void evaluerEtudiants()
    }
    Etudiant "1..*" -- "0..*" Cours : suit
    Professeur "1" -- "1..*" Cours : enseigne

3. Diagramme de séquence

Le diagramme de séquence illustre les interactions entre objets dans un scénario particulier du système.

Exemple concret : Processus de commande en ligne 

sequenceDiagram
    participant Client
    participant SiteWeb
    participant Système de paiement
    participant Entrepôt

    Client->>SiteWeb: Passer commande
    SiteWeb->>Système de paiement: Vérifier paiement
    Système de paiement-->>SiteWeb: Paiement confirmé
    SiteWeb->>Entrepôt: Préparer commande
    Entrepôt-->>SiteWeb: Commande prête
    SiteWeb-->>Client: Confirmation de commande

4. Diagramme d'activité

Le diagramme d'activité représente le flux de travail ou le processus d'un système.

Exemple concret : Processus de développement logiciel 

graph TD
    A[Début] --> B[Analyse des besoins]
    B --> C[Conception]
    C --> D[Implémentation]
    D --> E[Tests]
    E --> F{Tests réussis?}
    F -->|Non| D
    F -->|Oui| G[Déploiement]
    G --> H[Fin]

5. Diagramme d'état

Le diagramme d'état montre les différents états d'un objet et les transitions entre ces états.

Exemple concret : États d'une commande en ligne 

stateDiagram-v2
    [*] --> En_attente
    En_attente --> Validée: Paiement reçu
    Validée --> En_préparation: Traitement commencé
    En_préparation --> Expédiée: Envoi effectué
    Expédiée --> Livrée: Réception confirmée
    Livrée --> [*]

6. Exemple : Système de distributeur automatique de billets (DAB)

Voici un exemple détaillé d'un système de distributeur automatique de billets (DAB) utilisant tous les diagrammes UML précédemment mentionnés.

6.1. Diagramme de cas d'utilisation

Ce diagramme montre les interactions entre les utilisateurs et le système DAB. 

graph TD
    A[Client] -->|Retirer de l'argent| B((DAB))
    A -->|Consulter le solde| B
    A -->|Changer le code PIN| B
    C[Technicien] -->|Maintenir le DAB| B
    D[Banque] -->|Gérer les comptes| B

6.2. Diagramme de classes

Ce diagramme montre la structure statique du système DAB. 

classDiagram
    class DAB {
        -int id
        -float soldeDisponible
        +retirerArgent()
        +verifierSolde()
        +changerPIN()
    }
    class Compte {
        -int numeroCompte
        -float solde
        +debiter()
        +crediter()
        +verifierSolde()
    }
    class Client {
        -int id
        -String nom
        -int codePIN
        +authentifier()
    }
    class Transaction {
        -int id
        -Date date
        -float montant
        +enregistrer()
    }
    DAB "1" -- "1..*" Compte : gère
    Client "1" -- "1..*" Compte : possède
    DAB "1" -- "0..*" Transaction : effectue

6.3. Diagramme de séquence

Ce diagramme illustre le processus de retrait d'argent. 

sequenceDiagram
    participant Client
    participant DAB
    participant Compte
    participant Transaction

    Client->>DAB: Insérer carte
    DAB->>Client: Demander PIN
    Client->>DAB: Entrer PIN
    DAB->>Compte: Vérifier PIN
    Compte-->>DAB: PIN valide
    DAB->>Client: Demander montant
    Client->>DAB: Entrer montant
    DAB->>Compte: Vérifier solde
    Compte-->>DAB: Solde suffisant
    DAB->>Compte: Débiter montant
    DAB->>Transaction: Enregistrer transaction
    DAB->>Client: Distribuer argent

6.4. Diagramme d'activité

Ce diagramme représente le flux de travail du processus de retrait. 

graph TD
    A[Début] --> B[Insérer carte]
    B --> C[Entrer PIN]
    C --> D{PIN correct?}
    D -->|Non| E[Afficher erreur]
    E --> C
    D -->|Oui| F[Choisir opération]
    F --> G[Entrer montant]
    G --> H{Solde suffisant?}
    H -->|Non| I[Afficher solde insuffisant]
    I --> F
    H -->|Oui| J[Distribuer argent]
    J --> K[Imprimer reçu]
    K --> L[Fin]

6.5. Diagramme d'état

Ce diagramme montre les différents états du DAB. 

stateDiagram-v2
    [*] --> Inactif
    Inactif --> EnAttente: Carte insérée
    EnAttente --> Authentifié: PIN correct
    Authentifié --> TraitementTransaction: Opération sélectionnée
    TraitementTransaction --> DistributionArgent: Transaction validée
    DistributionArgent --> Inactif: Argent retiré
    Authentifié --> Inactif: Annulation/Timeout
    TraitementTransaction --> Authentifié: Transaction échouée

Cet exemple complet du système de distributeur automatique de billets (DAB) illustre comment les différents diagrammes UML peuvent être utilisés ensemble pour modéliser un système complexe sous différents angles. Chaque diagramme apporte une perspective unique sur le fonctionnement et la structure du système.

7. Exemple : Système de gestion de formation numérique

Voici un exemple détaillé d'un système de gestion de formation numérique utilisant les diagrammes UML pour illustrer les différents aspects du système.

7.1. Diagramme de cas d'utilisation

Ce diagramme montre les interactions entre les différents acteurs et le système de formation numérique. 

graph TD
    A[Étudiant] -->|Consulte l'agenda| B((Système de formation))
    A -->|Accède aux outils logiciels| B
    A -->|Soumet des travaux| B
    C[Professeur] -->|Planifie les cours| B
    C -->|Évalue les travaux| B
    D[Administrateur] -->|Gère les comptes| B
    D -->|Configure les outils| B
    E[Professionnel invité] -->|Participe à des sessions| B

7.2. Diagramme de classes

Ce diagramme représente la structure statique du système de formation numérique. 

classDiagram
    class Utilisateur {
        -int id
        -String nom
        -String email
        +seConnecter()
        +seDeconnecter()
    }
    class Etudiant {
        +consulterAgenda()
        +accederOutils()
        +soumettreTravail()
    }
    class Professeur {
        +planifierCours()
        +evaluerTravail()
    }
    class Administrateur {
        +gererComptes()
        +configurerOutils()
    }
    class Cours {
        -int id
        -String titre
        -Date date
        +ajouterParticipant()
        +supprimerParticipant()
    }
    class OutilLogiciel {
        -int id
        -String nom
        -String version
        +lancer()
        +fermer()
    }
    Utilisateur <|-- Etudiant
    Utilisateur <|-- Professeur
    Utilisateur <|-- Administrateur
    Etudiant "1..*" -- "0..*" Cours : participe
    Professeur "1" -- "1..*" Cours : enseigne
    Cours "1" -- "0..*" OutilLogiciel : utilise

7.3. Diagramme de séquence

Ce diagramme illustre le processus de connexion et d'accès aux outils de formation. 

sequenceDiagram
    participant U as Utilisateur
    participant S as Système
    participant A as Agenda
    participant O as OutilLogiciel

    U->>S: Se connecter
    S->>U: Demander identifiants
    U->>S: Fournir identifiants
    S->>S: Vérifier identifiants
    S->>U: Connexion réussie
    U->>S: Demander agenda
    S->>A: Récupérer agenda
    A-->>S: Renvoyer agenda
    S->>U: Afficher agenda
    U->>S: Demander accès outil
    S->>O: Vérifier autorisation
    O-->>S: Autorisation confirmée
    S->>O: Lancer outil
    O-->>U: Outil prêt à l'utilisation

7.4. Diagramme d'activité

Ce diagramme représente le flux de travail pour un étudiant participant à un cours en ligne. 

graph TD
    A[Début] --> B[Se connecter au système]
    B --> C[Consulter l'agenda]
    C --> D[Sélectionner un cours]
    D --> E[Accéder aux ressources du cours]
    E --> F[Lancer les outils nécessaires]
    F --> G[Participer au cours]
    G --> H[Réaliser les exercices]
    H --> I[Soumettre le travail]
    I --> J[Se déconnecter]
    J --> K[Fin]

7.5. Diagramme d'état

Ce diagramme montre les différents états d'une session utilisateur dans le système de formation. 

stateDiagram-v2
    [*] --> NonConnecte
    NonConnecte --> EnAttente: Saisie identifiants
    EnAttente --> Authentifie: Identifiants valides
    EnAttente --> NonConnecte: Identifiants invalides
    Authentifie --> ConsultationAgenda: Accès agenda
    Authentifie --> UtilisationOutil: Lancement outil
    ConsultationAgenda --> UtilisationOutil: Sélection cours
    UtilisationOutil --> SoumissionTravail: Fin exercice
    SoumissionTravail --> Authentifie: Travail soumis
    Authentifie --> NonConnecte: Déconnexion
    NonConnecte --> [*]
